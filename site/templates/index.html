Concrete

Systems code whose safety claims are inspectable.

Concrete is an experimental systems language where authority, runtime risk, proof evidence, assumptions, and trusted boundaries are visible in source and replayable through reports.

Thesis

Safety claims you can review, replay, and challenge.

Concrete is built around explicit accounting: what a function can touch, what was proved, what was only tested, what must be trusted, and what remains assumed are separate facts instead of one green badge.

Credibility Slice

A narrow slice you can inspect and replay.

The production-readiness path is a real parser, codec, or crypto core with its source, contracts, capability report, oracle tests, proof status, and trusted boundaries shown together.

Code Shape

Concrete code keeps cost, authority, and evidence visible.

The syntax is intentionally ordinary. The important part is what the surface refuses to hide: capabilities, checked obligations, value access, and scoped borrowing.

Contracts and authority

#[overflow_checked]
#[requires(0 <= sample && sample <= 255)]
fn scale(sample: i32, gain: i32) -> i32 {
    return (sample * gain) / 256;
}

pub fn main() with(Console) -> i32 {
    Console::print_i64(scale(120, 200) as i64);
    return 0;
}

Reports can say which obligations were proved, which were checked, and that this entry point needs Console authority.

Value-model collection access

let value: Option<i32> = counts.get(key);

let present: bool = counts.update(&key, bump);
if !present {
    counts.insert(key, 1);
}

let seen: Option<bool> =
    names.with_value(&id, &needle, same_name);

Copy reads return values; mutation is an operation; non-Copy borrowed reads stay scoped inside with_value.

Parser-friendly patterns

match byte {
    0x00..=0x1f => return Token::Control,
    0x30..=0x39 => return Token::Digit,
    b if b == quote => return Token::Quote,
    _ => return Token::Other,
}

if let Result::Ok { value } = read_header(c) {
    return value.len as i32;
}

Pattern ergonomics target real decoder/parser code while keeping exhaustiveness and lowering behavior gated.

Evidence Model

Concrete reports the strength of a claim.

A Lean theorem, a kernel decision, an oracle test, and an assumption are different things. Concrete keeps them different in the source, ledger, reports, and gates.

Class Means Concrete surface Not claiming
proved_by_lean Checked by Lean’s kernel through a linked theorem. #[proof_by(...)], proof fingerprints, stale gates. Backend or machine timing correctness.
proved_by_kernel_decision Discharged by built-in decision procedures. Bounds, div/mod, overflow, loop arithmetic obligations. A general theorem prover for arbitrary specs.
enforced Rejected or constrained by the compiler. Capabilities, no returned refs, value model, provenance rules. A proof that the program does the intended job.
runtime_checked Kept safe by generated or library checks. Checked accessors, fallible parsers, explicit policy reports. A static proof.
tested_by_oracle Compared against an independent implementation or corpus. Execution oracles, snapshot tests, differential fixtures. Complete correctness.
assumed / trusted Outside the proved model and named for audit. Trusted std internals, backend assumptions, environment facts. Something hidden under a “safe” label.

Language Shape

Small rules with audit consequences.

Explicit authority

Functions carry capabilities, so allocation, IO, process access, and unsafe authority are reviewable facts.

Value model

Copy, move, borrow, and future clone are distinct API cells. A map read does not hide a clone or escaping reference.

No returned refs

Safe-callable functions cannot return references. Borrowed data flows downward through scoped callbacks and blocks.

Contracts in source

Preconditions, postconditions, runtime-safety obligations, and proof links stay next to the code they describe.

Parser ergonomics

Ranges, guards, OR patterns, if let, while let, match-on-reference, and struct update serve decoder workloads.

ByteView

Stored zero-copy byte views are coordinates, not references, so they fit the no-lifetimes memory model.

Positioning

Concrete sits between systems control and proof accounting.

The point is not to beat mature languages on their home ground. It is to make a different trade: smaller systems-language surface, explicit authority, and replayable evidence in one workflow.

Rust

Rust is the maturity benchmark for ownership, libraries, and tooling. Concrete is smaller and less mature, but makes authority and evidence part of the language reports instead of external process.

Zig

Zig is excellent at explicit low-level control and build ergonomics. Concrete keeps the control story small, then adds contracts, proof status, and capability accounting.

SPARK / Ada

SPARK has the strongest production assurance precedent. Concrete borrows the discipline but aims for a modern systems surface with ownership, capabilities, and explicit trusted boundaries.

Lean / F*

Lean and F* can express deeper proofs. Concrete keeps ordinary systems code central, uses Lean where it pays, and pushes common operational obligations toward automation.

C + analyzers

C plus static analyzers is still the deployed baseline. Concrete’s bet is that the evidence ledger and capability surface should be native, not bolted on after the fact.

Dafny / Why3

Verifier-first languages prove more by default. Concrete accepts a narrower proof story to stay close to systems workloads and explicit runtime authority.

Status

Strong experimental core, not a production promise.

Solid today

  • Phases 1-5 core closed or workload-deferred.
  • Phase 6 formatter, aliases, loop control, and pattern ergonomics gated.
  • H1 returned-reference class closed by the type rule.
  • Obligation ledger is the single source for major reports.

Still being built

  • Numeric literal and cast policy.
  • Defer and cleanup ergonomics.
  • Operational VC automation for fewer hand-written Lean bridges.
  • Certification-style release bundles and production examples.

Deliberately deferred

  • Const generics until a real capacity-parametric workload pulls them.
  • Clone and mutable scoped callbacks until their value-model rules are forced.
  • Device capability until freestanding hardware code exists.
  • from(param) only if scoped-borrow APIs prove insufficient.
