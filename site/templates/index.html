Concrete

Linear systems code with evidence you can audit.

Concrete is an LL(1), linear, capability-oriented systems programming language designed for auditable and formally verifiable code. Non-Copy values are used exactly once, function headers expose required capabilities, and the compiler/tooling preserve a clear path from source code to evidence.

Linear by default

Copy is the explicit escape for ordinary duplicable data. Everything else is used exactly once: moved, consumed, returned, or destroyed explicitly.

Authority in headers

A function that allocates, prints, reads files, uses unsafe code, or crosses a trusted boundary has to say so in its signature and reports.

Proof without pretending

Lean proofs, kernel decisions, SMT runs, oracle tests, runtime checks, and assumptions are reported as different evidence classes.

Code Shape

Small systems code, explicit obligations.

Concrete keeps the source ordinary enough to read, but makes authority, ownership, and proof obligations visible instead of relying on comments or review folklore.

Pure code with runtime obligations

#[requires(0 <= off && off + 1 < len && len <= 512)]
fn read_u16_be(packet: [u8; 512], off: i32, len: i32) -> i32 {
    let hi: i32 = packet[off] as i32;
    let lo: i32 = packet[off + 1] as i32;
    return hi * 256 + lo;
}

Effects are part of the type surface

fn validate(data: i32, len: i32) -> i32 {
    if len < 10 { return 1; }
    return data + len;
}

fn report(result: i32) with(Console) {
    if result == 0 { println("ok"); }
    else { println("fail"); }
}

Language Model

The rules are intentionally small.

01

LL(1) surface

The grammar is meant to stay predictable and mechanically checked, so the language remains easy to parse, format, and audit.

02

Linear ownership

Non-Copy values cannot be duplicated or silently discarded. Resource handling stays explicit.

03

Capability flow

Authority is visible in function headers and can be traced through reports and call graphs.

04

Checked runtime risk

Bounds, arithmetic traps, division, assertions, and preconditions become checks or proof obligations.

05

Named trust

Unsafe code, backend assumptions, FFI, SMT trust, and environment assumptions appear as named boundaries.

06

Lean substrate

The compiler is written in Lean 4, and selected source-linked user claims can be checked by Lean's kernel.

Evidence Model

No single green badge.

Concrete’s reports distinguish what was proved, what was checked at runtime, what was tested, what was assumed, and what depends on a trusted boundary.

proved_by_lean A linked theorem checked by Lean's kernel.
proved_by_kernel_decision A generated obligation closed by Lean-owned decision procedures.
runtime_checked A dynamic check remains in the program or instrumentation.
tested_by_oracle Compiled behavior matched an independent implementation or corpus.
assumed / trusted The claim crosses a named boundary instead of disappearing into prose.
unproven / stale The obligation exists, or the linked proof no longer matches the source.

Positioning

Built between systems languages and proof tools.

Rust

Concrete borrows the seriousness about ownership, but stays linear rather than affine and makes authority/proof evidence part of reports.

Zig and Odin

Concrete shares the taste for explicit low-level control, while rejecting ambient authority and adding a formal evidence ledger.

SPARK, Dafny, F*

Concrete learns from verification-first systems, but keeps a small C/Rust-shaped systems surface and separates every evidence class.

Austral

Concrete is closest in linearity, capabilities, no GC, and no implicit cleanup, then adds Lean-backed source evidence and audit reports.

Try It

Start with the gates.

Concrete is experimental. The most honest first contact is to build it, run the focused gates, and inspect the reports. 

$ make build
$ make test
$ make test-linear-conservation
$ make test-linear-discard
$ make check-grammar
$ .lake/build/bin/concrete examples/parse_validate/src/main.con --report effects
$ .lake/build/bin/concrete examples/vc_suite/fixed_point_filter.con --report vcs
$ .lake/build/bin/concrete examples/constant_time_tag/src/main.con --report audit
