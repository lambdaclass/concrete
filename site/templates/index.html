Concrete

Experimental systems language with Lean-backed evidence.

Concrete is a small no-GC systems language with linear ownership, explicit capabilities, and a proof/audit pipeline built around Lean. The project is not a verified compiler today. It is an experiment in keeping source code, authority, assumptions, tests, and proofs tied together under normal development.

What exists today

Concrete tracks evidence as project data, not prose.

enforced

Authority and allocation reports

Reports name capabilities, allocation, trusted code, unsafe use, and proof eligibility per function.

Lean

Attached user theorems

Proofs are checked by Lean and tied to source fingerprints through the proof registry.

drift gate

Stale proof detection

If the source body or registered spec changes, proof status downgrades instead of staying green.

tested

Oracles and snapshots

Flagships carry differential tests, policy files, assumptions, release bundles, and snapshot baselines.

partial

Extraction soundness

Some source-to-ProofCore rules are proved; the compiler and LLVM backend remain in the trusted base.

planned

Source contracts and SMT

Contracts, VC generation, SMT assistance, and ghost code are design work, not shipped guarantees.

Example

A real proof-bearing function.

Concrete source constant_time_tag.ct_compare 
fn ct_compare(a: [u8; 16], b: [u8; 16]) -> Int {
    let diff: u8 = 0;
    let i: Int = 0;

    while i < 16 {
        set diff = diff | (a[i] ^ b[i]);
        set i = i + 1;
    }

    if diff == 0 { return 1; }
    return 0;
}

This is ordinary systems-shaped code: fixed arrays, byte operations, no allocation, no ambient authority.

Audit output evidence stays classified 
$ concrete audit examples/constant_time_tag/src/main.con

proof:
  ct_compare_same_tag_correct: proved_by_lean
  coverage: one_direction

authority:
  capabilities: none
  allocation: none
  trusted code: none

assumptions:
  machine-level constant-time: assumed
  backend timing preservation: trusted

The report does not turn all evidence into one badge. It separates Lean-proved facts from assumptions and trusted backend behavior.
