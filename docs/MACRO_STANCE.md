# Macro / Metaprogramming Stance

Status: PERMANENT NON-GOAL — ROADMAP Phase 6 #11.
Gated by `scripts/tests/check_no_macros.sh`.
Date: 2026-06-23

## Decision

Concrete has **no language macro system**.

Rejected permanently:

- `macro` / `macro_rules`-style declarations;
- procedural macros;
- bang-invocation syntax such as `foo!(...)`;
- `#[derive(...)]` or derive-like helpers;
- syntax macros;
- compile-time source generation inside the language;
- arbitrary `comptime` evaluation as a code-generation mechanism.

This is not a v1 omission. It is part of Concrete's audit model: the code a
reviewer reads is the code that typechecks, lowers, produces obligations, and
appears in reports. Hidden expansion would obscure capabilities, allocation,
trusted boundaries, generated obligations, source spans, and proof evidence.

## What Concrete Uses Instead

- Ordinary functions.
- Monomorphized generics.
- Explicit capability-polymorphic function pointers.
- Named structs and enums instead of generated product/sum boilerplate.
- Small compiler-recognized attributes that classify contracts, policy, or
  evidence, but do not generate source.
- External build-time generators only if their generated files are checked in or
  otherwise audited as ordinary source artifacts.

External generators are tools, not language features. Their output must be
visible to formatter, diagnostics, reports, gates, and source review.

## Why No Derive Exception

`#[derive(...)]` looks smaller than a macro system, but it has the same audit
failure mode: source that is not visibly written by the programmer becomes part
of the checked program. Equality, debug/display, serializers, parsers, proof
stubs, tables, and contract boilerplate must be explicit source or generated by
an external tool whose output is reviewed like any other file.

## Gate

`scripts/tests/check_no_macros.sh` locks the negative surface:

- macro declarations are rejected;
- bang invocations are rejected;
- derive-like attributes are rejected;
- no compiler path may introduce hidden source expansion.

If a future workload wants code generation, the proposal must be for an external
generator or ordinary library API first. Reopening language macros requires a
new thesis-level decision, not a routine roadmap item.
