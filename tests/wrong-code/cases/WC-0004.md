# WC-0004 — accumulator-style match triggers SSA-verify dominator violation

**Category:** verifier-trigger
**Status:** open
**Discovered:** parse_validate phase-3 work (see CHANGELOG note on Phase 3 parse_validate landing — originally reported as E0703)
**Fixed:** —

## Symptom

A function with several sequential `match` blocks plus a mutable
accumulator counter — the natural shape for "validate a record by
running each check, incrementing a failure count" — triggers an
`SSAVerify` dominator error. Rewriting the same logic in early-return
style compiles cleanly and the runtime behavior is identical.

`parse_validate` was originally written in accumulator style and had
to be rewritten to early-return form to ship. The underlying compiler
bug was never minimized or fixed; the workaround is in the example.

The candidate minimized repro at `cases/WC-0004/program.con` triggers
`E0708` against the current compiler:

```
error[ssa-verify]: (E0708) count_failures: block 'merge5': phi operand
%t9 from 'arm06' not dominated by def block 'arm13'
```

The original CHANGELOG note cited `E0703`. Both codes are dominator
violations in `Concrete/SSAVerify.lean` (E0703 is "use in non-
dominating block"; E0708 is "phi operand from src not dominated by
def block"). They are the same family of bug; the verifier appears to
have been refactored between the original report and now, narrowing
the diagnostic to a more specific code.

## Root cause

Unknown. Symptom suggests the SSA emission path for sequential
`match` interacts incorrectly with mutable-counter promotion across
match blocks. The dominator check fires on a phi operand whose source
value is not actually dominated by the predecessor in the
`match`-arms-merged shape.

## Fix

None. The case is open. A successful fix likely lives in
`Concrete/SSAVerify.lean` (the dominator computation) or
`Concrete/Lower.lean`'s match lowering, but the bug must be minimized
first.

## Minimization

The repro at `tests/wrong-code/cases/WC-0004/program.con` is the
smallest accumulator-style match that still triggers the dominator
violation. It needs three sequential `match` arms over a `Pass{}/
Fail{...}` enum, each writing to one shared `mut failures: i32`
counter, with no early returns. Removing any of those — fewer arms,
single-arm match, immutable counter, early return inside an arm —
collapses the program into a shape the verifier accepts.

The case was originally reported as E0703 against an earlier verifier;
the current code path narrows to E0708 (phi-source not dominated by
def). The bug is the same family: SSA emission for sequential `match`
+ mut-counter promotion produces a phi whose operand is defined in a
block that does not dominate the predecessor edge it arrives on.

A successful fix lives somewhere in the path between
`Concrete/Lower.lean`'s match lowering (which decides where the
counter assignment block lands relative to merge points) and
`Concrete/SSAVerify.lean`'s dominator construction (which may be
overly strict for the merged shape). Bisect from the simpler end:
make the verifier accept the existing emission, then verify nothing
downstream depends on the rejected shape.

## How to verify it stays fixed

`make test-wrong-code` runs the manifest entry: while `status =
"open"`, the script asserts the verifier *still* rejects the program
with E0708, so we notice immediately if the bug silently fixes
itself (which would mean the manifest line needs to flip to `fixed`)
or if it stops being a verifier error (which would mean it became a
miscompile and needs a different category).

When the bug is fixed:

- Set `status = "fixed"`.
- Change `kind` to `runtime` and set `expected` to the runtime value
  (the candidate is built so `count_failures(50)` returns the count
  of failed checks for input 50, which should be deterministic).
- Add the fix commit to `fixed`.
- Update this notes file with which pass actually held the bug.
