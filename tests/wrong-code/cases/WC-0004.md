# WC-0004 — match arm-local bindings leaked into next match's var-table

**Category:** miscompile (caught by SSA-verify dominator check before
codegen — would have produced wrong code if the verifier had been
less strict)
**Status:** fixed
**Discovered:** parse_validate phase-3 work (CHANGELOG entry —
original code E0703, candidate-minimization symptom E0708)
**Fixed:** WC-0004 fix in `Concrete/Lower.lean` match merge

## Symptom

A function with two or more sequential `match` blocks over an enum
that has a payload-binding variant (e.g. `Check::Fail { code }`),
plus a mutable accumulator variable shared across the matches,
triggers `SSAVerify` error E0708:

```
error[ssa-verify]: (E0708) <fn>: block 'merge<N>': phi operand %tX
from 'arm<a>' not dominated by def block 'arm<b>'
```

`parse_validate` was originally written in this shape (an
accumulator `failures` counter incremented inside the `Fail` arm of
each per-field validation match) and was rewritten to early-return
style to ship. The early-return rewrite worked because each `Fail`
arm in early-return form ends with `return`, so it doesn't reach the
post-match merge — and the leak path stays out of the var-table for
the next match.

The original symptom report cited E0703 (use-in-non-dominating-
block); the current candidate triggers E0708 (phi-source-not-
dominated). Both are dominator violations in
`Concrete/SSAVerify.lean`. The verifier was correctly catching wrong
SSA emission; the bug was upstream in `Concrete/Lower.lean`.

## Root cause

`Lower.lean`'s `.match_` lowering calls `setVar bname (.reg loadDst
bty)` to bind enum-payload fields like `code` inside each arm body.
`setVar` mutates the global var-table.

Before each arm, the lowering restores `vars := preMatchVars`, so
arms don't see each other's bindings. **But the merge logic at the
end did not undo this**: after the final arm runs, the var-table
still carried whatever bindings that arm introduced.

`snapshotVars` then captured this leaked state when the next match
called `let preMatchVars ← snapshotVars` for *its* preMatchVars.
That second-match preMatchVars therefore included the arm-local
binding `code` from the prior match. The next match's merge logic
iterated over those leaked names and built a phi for them — but
only some arms had `code` bound (the `Fail` arm), and the arms that
didn't (the `Pass` arm) carried the *previous match's* `code`
register.

The result: `phi i32 [%t9, %arm06], [%t21, %arm18]` where `%t9` is
defined inside the previous match's `Fail` arm, which doesn't
dominate the current match's `Pass` arm. Verifier rejection.

The same pattern would have produced wrong runtime behavior if the
verifier had been less strict: the phi would silently take whatever
register the upstream `Fail` arm happened to define, regardless of
which arm the current match took.

## Fix

`Concrete/Lower.lean`, after the match's variable-merge logic:
restrict the var-table back to the names that existed in
`preMatchVars`. The merge has already updated those entries with
their phi registers (or `match.merge` allocas for aggregates); the
leaked arm-local bindings just need to be dropped.

The fix applies in all three exit paths — `liveSnapshots.length >=
2` (normal multi-arm merge), `liveSnapshots.length == 1` (only one
arm reaches merge; previously copied the whole arm-end vars
including bindings), and the all-terminated case (merge unreachable
but downstream code might still snapshot).

## Minimization

The repro at `tests/wrong-code/cases/WC-0004/program.con` is the
smallest accumulator-style match that triggers the dominator
violation. It needs:

- An enum with at least two variants where at least one binds a
  field (e.g., `Fail { code: i32 }`).
- Two or more sequential `match` expressions over that enum.
- A mutable counter shared across the matches and modified inside
  the field-binding arm.

Removing any of those — fewer arms, single-arm match, immutable
counter, no payload field, single match — collapses the program
into a shape that doesn't exercise the leak.

The candidate program runs three checks against `count_failures(50)`
(checks for `> 0`, `< 100`, `!= 42` — all pass for input 50), so
the expected output is `0`. A runtime regression here would surface
either as the verifier catching a re-introduction of the leaked
phi or, if the verifier ever weakens, as a wrong runtime value.

## How to verify it stays fixed

`make test-wrong-code` runs the manifest entry; the binary must
print `0`.

A new diagnostic plumbing — `concrete <file> --emit-ssa-unverified`
plus `Pipeline.lowerUnverified` — was added to inspect raw SSA
output that the verifier rejects. It bypasses both the verifier and
the cleanup pass. It is debugging infrastructure for the wrong-code
workflow (per `docs/WRONG_CODE_CORPUS.md` artifact bundle), not part
of the production compile path.
