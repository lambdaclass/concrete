# WC-0007 — arrow-assign narrow field elaborated RHS as `Int`

**Category:** miscompile
**Status:** fixed
**Discovered:** 38ce7f5 (sibling family of WC-0003)
**Fixed:** 38ce7f5

## Symptom

`p->n = 100` where `n: i32` (and `p` is a pointer-typed receiver)
elaborated `100` as `Int` (i64). Codegen emitted `store i64 100` to a
4-byte field; clang `-O2` deleted the UB store and the field stayed
at its initial value, while the *adjacent* field was clobbered.

Sibling to WC-0003 (the field-assign narrow-field bug). The two paths
were elaborated separately in `Concrete/Elab.lean`; the
`fieldAssign` path got the type-hint fix first, the `arrowAssign`
path got it later.

## Expected behavior

`tests/programs/bug_arrow_assign_narrow_field.con` returns `107`
(struct shape: `Pair { a: i32 = 100, b: i32 = 7 }`; the test reads
both fields back to confirm the adjacent field was not clobbered).

## Regression command

`make test-wrong-code`.

## Historical context

Fix landed in `38ce7f5`. The pair WC-0003 / WC-0007 are an example of
a class-of-bug that needs to be fixed everywhere it's structurally
possible: any time `Elab` calls `elabExpr value` without a field-type
hint into a narrow numeric field, the same UB is reproducible.
