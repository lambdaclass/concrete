# WC-0001 — chained newtypes broke the cast-validity exemption

**Category:** miscompile
**Status:** fixed
**Discovered:** 66d8736 (adversarial sweep)
**Fixed:** 66d8736

## Symptom

`Outer = Middle = Inner = i32` rejected `Middle(Inner(...))` at E0553.
Constructing a chained-newtype value should be a representation no-op
(every layer is the same `i32` underneath); the type checker rejected
it instead.

## Root cause

`Concrete/CoreCheck.lean`'s cast-validity exemption checked whether
one side of a `.cast` resolved to the same primitive as the other,
using `Layout.resolveNewtype` — which recurses all the way to the
underlying primitive. The constructor wraps one step at a time
(`Inner` → `Middle` → `Outer`), so the exemption needed a one-step
unwrap, not a full resolve.

## Fix

`Concrete/CoreCheck.lean`: replaced the recursing check with a
`oneStepInner` helper that finds the immediate inner type only, and
tries both wrap and unwrap directions independently.
`Layout.ntSubstTy` was promoted from `private` so CoreCheck can reuse
it for generic newtypes.

## Minimization

The repro at `tests/programs/adversarial/newtype/chained.con` is
already minimal: three newtypes stacked one-deep each, one constructor
chain, one method dispatch through the layers. Removing any layer
collapses the bug into a one-step newtype case which the original
exemption *did* handle.

## How to verify it stays fixed

`make test-wrong-code` runs the manifest entry; the binary must print
`42`.
