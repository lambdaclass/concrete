# WC-0003 — narrow-int field assign elaborated the RHS as `Int`

**Category:** miscompile
**Status:** fixed
**Discovered:** 66d8736 (adversarial sweep)
**Fixed:** 66d8736

## Symptom

`c.n = 100` where `n: i32` elaborated `100` as `Int` (i64) and codegen
emitted `store i64 100` to a 4-byte field — undefined behavior. clang
`-O2` deleted the UB store and the field stayed at its initial value;
`lli` happened to round-trip the low 4 bytes, so the bug only surfaced
when reading back via the native `-O2` binary.

This is the kind of bug that shows up nowhere except production: it
lived inside a single store instruction, the optimizer was acting on
its license to assume the program was UB-free, and the LLVM
interpreter got the "right" answer by accident.

## Root cause

`Concrete/Elab.lean`'s `.fieldAssign` path called `elabExpr value`
without a type hint. With no hint, integer literals default to `Int`
(i64), so `100` was an i64 even though the field was i32. The
resulting cast was nominally there but codegen was seeing the wrong
operand type.

## Fix

`Concrete/Elab.lean` `.fieldAssign` path: look up the field's declared
type from the struct definition and pass it as the value hint to
`elabExpr`, so `100` is elaborated as `i32` directly.

## Minimization

`tests/programs/bug_field_assign_narrow_field.con` is one struct with
one i32 field, one assignment, one read-back. Removing the read-back
hides the bug behind UB elimination; removing the assignment makes the
program trivial. The minimal form needs exactly: a narrow integer
field, an integer-literal RHS, an assignment, and an observation.

## How to verify it stays fixed

`make test-wrong-code` runs the manifest entry; the binary must print
`100`. A regression here would manifest as `0` or another residual
value depending on the optimization level.
