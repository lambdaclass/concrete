# WC-0002 — `.0` on a borrowed newtype produced `&Newtype → Inner`

**Category:** miscompile
**Status:** fixed
**Discovered:** 66d8736 (adversarial sweep)
**Fixed:** 66d8736

## Symptom

`&self.0` (and `&mut self.0`) inside an inherent impl method emitted
a `.cast` from `.ref Newtype` to the inner primitive. CoreCheck's
exemption (matching only direct `Newtype ↔ Inner` pairs) rejected it,
and codegen would have mishandled the ref → value transition.

## Root cause

`Concrete/Elab.lean`'s field-access path treated newtype `.0` access
uniformly regardless of whether the receiver was the value or a
borrow. When the receiver was a ref/refMut, the cast that rebrands
`Newtype` to its inner type was still emitted, but the receiver hadn't
been deref'd, so the cast's source was `&Newtype` and the destination
was the bare inner primitive.

## Fix

`Concrete/Elab.lean` field-access path: when the receiver type is
`.ref _` or `.refMut _`, emit a `.deref cObj newtypeTy` first so the
rebrand cast is `Newtype → Inner` as expected.

## Minimization

`tests/programs/adversarial/newtype/borrow.con` exercises `&self`,
`&mut self`, and `self` instance-methods on a single newtype, plus
`.0` access across the borrow boundary. Each method form is needed:
removing any of them collapses the bug into a no-deref-needed case
that the pre-fix code accidentally handled.

## How to verify it stays fixed

`make test-wrong-code` runs the manifest entry; the binary must print
`8080`.
